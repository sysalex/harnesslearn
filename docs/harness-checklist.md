# Harness 审查清单

> 在将 Harness 部署到生产环境或交接前运行此清单。
> 失败项 = 阻塞项；跳过项需要书面理由。

## Agent 指令（AGENTS.md）

- [ ] 项目概述准确且最新
- [ ] 仓库结构反映当前布局
- [ ] 工具权限明确指定（允许/限制/禁止）
- [ ] 验证门禁已定义且命令正确
- [ ] 无模糊指令（不会有多种解释）

## 工具设计

- [ ] 每个工具有清晰、无歧义的名称
- [ ] 工具 schema 最小化（无 agent 不使用的可选字段）
- [ ] 错误消息告诉 agent 下一步做什么，而不仅是哪里出错
- [ ] 工具返回值一致（成功和失败时形状相同）
- [ ] 无工具做超过一个概念性的事情

## 上下文传递

- [ ] 上下文限定在 agent 任务所需范围（不是整个代码库）
- [ ] 长期状态（计划、决策、进度）在文件中，不在 prompt 中
- [ ] 为多会话任务定义上下文压缩策略
- [ ] agent 可访问的上下文中无敏感数据（密钥、凭证）

## 规划产物

- [ ] 非平凡任务存在 PLAN.md
- [ ] 里程碑有明确的验证命令
- [ ] 范围边界（范围内/范围外）已写明
- [ ] IMPLEMENT.md 捕获决策和偏离

## 权限和沙箱

- [ ] Agent 以任务所需的最小权限运行
- [ ] 破坏性操作需要明确确认
- [ ] 网络访问尽可能限定范围
- [ ] 文件系统访问限定在项目目录内

## 验证循环

- [ ] 为 agent 的输出编写了测试
- [ ] Agent 可以自己运行验证命令（不仅是"人工审查"）
- [ ] 验证在任务完成时自动运行，不仅在 PR 时
- [ ] 评估标准在任务开始前写下，不是事后

---

## 何时移除此 Harness 组件

> 每个 Harness 组件存在是因为模型还做不到某件事。
> 文档化什么能力改进可以让此组件变得不必要。

| 组件 | 存在原因 | 何时可以移除 |
|------|---------|-------------|
| AGENTS.md | Agent 需要明确的工作指令 | 模型能自动理解项目约定 |
| CLAUDE.md | 需要导航入口和架构约束 | 模型能从代码推断架构 |
| task-list.md | 需要任务追踪和进度管理 | 模型能自动管理任务状态 |
| definition-of-done.md | 需要明确的质量标准 | 模型能自动判断完成质量 |
| security-checklist.md | 需要安全检查清单 | 自动化安全扫描覆盖所有检查项 |
| feedback-loop.md | 需要四层反馈循环 | 模型能自动建立反馈机制 |
